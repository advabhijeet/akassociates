# Article Reusable Modules

Last updated: 2026-05-15

This document records the reusable article architecture for the Citadel article system currently used by Chambers of AK.

It should be read with:

```text
docs/maintenance/ARTICLE_PUBLISHING_WORKFLOW.md
docs/maintenance/ARTICLE_HTML_TEMPLATE.md
docs/planning/CHAMBERS_OF_AK_PLATFORM_V2_ROADMAP.md
```

## Core Rule

Article pages should contain the article content. Reusable article interface blocks should be generated by shared modules, not manually copied into every article file.

Do not manually hard-code the following blocks page-by-page unless a module is unavailable and a temporary exception is documented:

```text
- Article Index / table of contents;
- reading progress bar;
- Tags block;
- Previous Article / Next Article block;
- Recommended Reads block;
- future comments / feedback block;
- future article ad slots.
```

## Citadel Feature Scope

The reusable article UI system is a Citadel feature, not merely a Chambers of AK one-off implementation.

This means:

```text
- Article Index is a Citadel article module.
- Article Footer v2 must also be treated as a Citadel article module.
- The behaviour should be reusable across Citadel-powered article pages, not hardcoded only for Chambers of AK pages.
- Chambers-specific data can exist in the current registry, but module logic should remain portable.
- CSS and JavaScript selectors should prefer generic article/module classes where practical.
- Future extraction should move module behaviour under the Citadel theme/module structure instead of keeping all logic inside the global site script.
```

Current Chambers-specific registry name:

```text
window.chambersInsightsRegistry
```

Future portable direction:

```text
window.CitadelArticleRegistry or assets/data/insights-registry.json
```

The immediate website implementation can continue using the current Chambers registry, but Article Footer v2 should be written as a reusable Citadel module with fallback behaviour and minimal brand-specific assumptions.

## Article Index Module

Current module files:

```text
assets/css/themes/citadel-of-kang/modules/article-index.css
assets/js/themes/citadel-of-kang/article-index-direct-rail.js
```

Current behaviour:

```text
- desktop direct-rail article index;
- mobile static index below hero / before article body;
- mobile progress bar below navigation;
- drawer-safe progress hiding;
- active heading sync;
- click-jump offset below fixed navbar;
- no internal article-index scrollbar;
- no horizontal drift.
```

Target next improvement:

```text
The Article Index should auto-apply to all current and future article pages that use <article class="article-body"> and have enough h2 headings.
```

No future article should require manual wiring of the Article Index script if a safe global loader is implemented.

## Article Footer Module

Current module location:

```text
assets/js/script.js
```

Current module label:

```text
// Article footer: tags, previous/next and recommended reads
```

Current generated blocks:

```text
- Tags;
- Previous Article;
- Next Article;
- Recommended Reads.
```

Current dependency:

```text
window.chambersInsightsRegistry
```

Current weakness:

```text
If a page under /updates/ exists but is missing from window.chambersInsightsRegistry, the Article Footer module silently returns and no Tags / Previous / Next / Recommended Reads block appears.
```

This caused the property title search article to show Article Index but not the article-bottom Tags / Previous / Recommended Reads block.

## Article Footer v2 Target

The Article Footer must become a Citadel reusable feature that is fault-tolerant and auto-applied.

Required v2 behaviour:

```text
1. Detect article.article-body automatically.
2. Do nothing if an article footer already exists.
3. Try to find the current article in the available article registry.
4. If found, use registry metadata for tags, previous/next and recommended reads.
5. If not found, derive fallback metadata from:
   - h1 title;
   - meta description;
   - hero eyebrow / category label;
   - article-meta spans;
   - article data attributes, if present.
6. Render a useful footer instead of silently disappearing.
7. Never duplicate the footer if the module runs twice.
8. Keep the generated footer inside article.article-body at the end of the article content.
9. Keep module logic portable enough for Citadel-powered pages beyond Chambers of AK.
```

Suggested optional article attributes for stronger fallback:

```html
<article class="article-body"
  data-article-category="Checklist"
  data-article-tags="Property, Due Diligence, Title Search, Checklist">
```

These attributes should help fallback accuracy but should not become mandatory after the module is robust.

## Central Registry Rule

For the current static site, article metadata is centralised in:

```text
window.chambersInsightsRegistry inside assets/js/script.js
```

This registry powers:

```text
- homepage insight cards;
- Insights page filter/search/list rendering;
- category and tag chips;
- article thumbnails;
- article hero backgrounds;
- Article Footer tags;
- previous/next navigation;
- recommended reads.
```

Every public article under `updates/` should either:

```text
- exist in the registry; or
- have sufficient fallback metadata for Article Footer v2.
```

## Validation Requirement

Add or maintain a validation script that checks:

```text
- every /updates/*.html article with article-body is present in the registry;
- every registry href points to an existing file;
- every article has h1;
- every article has at least three meaningful h2 headings if Article Index is expected;
- no duplicate article-footer blocks are manually hardcoded;
- no duplicate Article Index scripts are inserted.
```

This validation should be run before major article batches and before any publishing workflow is treated as complete.

## Long-Term Manifest Direction

The registry should eventually move out of `assets/js/script.js` into a dedicated data file, for example:

```text
assets/data/insights-registry.json
```

Long-term separation:

```text
HTML article files = article content
assets/data/insights-registry.json = article metadata
assets/js/script.js = behaviour/modules
CSS/theme files = visual styling
```

This will make future article publishing, admin CMS integration and database migration easier.

Citadel long-term module separation should be:

```text
assets/js/themes/citadel-of-kang/article-index-direct-rail.js
assets/js/themes/citadel-of-kang/article-footer.js
assets/css/themes/citadel-of-kang/modules/article-index.css
assets/css/themes/citadel-of-kang/modules/article-footer.css
```

## Zero-Tolerance Rule

Do not solve reusable article UI problems by copying blocks into individual pages.

Correct approach:

```text
- improve the shared module;
- improve the registry/manifest;
- add fallback behaviour;
- add validation;
- document the rule.
```

Manual page-by-page insertion is allowed only as a temporary emergency fix and must be removed when the reusable module is stable.
